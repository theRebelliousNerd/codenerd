# North Star

## Vision Statement
The `cmd/query-kb` subsystem will become the standard diagnostic CLI for querying Knowledge Base databases across the NERD ecosystem without external dependencies.

## Strategic Goals

### Goal 1: Scalable Shard Discovery
Current discovery logic reads the `.nerd/shards` directory and checks every file extension. This approach scales poorly for large directory structures. The system must implement intelligent path resolution to bypass full directory scans. Success is measured by keeping discovery time under 50ms regardless of the file count.

### Goal 2: Bounded Vector Inspection
Vectors hold float arrays that overwhelm the terminal buffer when `deep_query.go` prints them. The system must add numerical truncation logic to the `metadata` string output. Success involves parsing vector arrays and summarizing their lengths instead of printing raw float strings.

### Goal 3: Zero-Dependency Architecture
The `cmd/query-kb` subsystem uses the `modernc.org/sqlite` package. This allows compilation without C compiler dependencies. The goal is to keep the binary free from CGo requirements while adding support for concurrent database access.

## Guiding Principles
1. **Safety First:** The system protects standard console outputs with `sync.Mutex` locks.
2. **Read-Only Guarantees:** The subsystem executes `SELECT` and `PRAGMA` queries; it must not execute `INSERT` statements outside of `main_test.go`.
3. **Graceful Degradation:** The code skips invalid SQLite schemas rather than calling `os.Exit`.

## Non-Goals
* Attempting to rebuild corrupted database files.
* Implementing a web interface for database administration.
* Generating full SQL dumps.

## Extended Documentation Notes

The above specification details represent the functional baseline for the system. Code definitions in `main.go` establish the connection loops.
The sub-routines located in `deep_query.go` handle the iterative data parsing.
Validation checks rely on testing logic found inside `main_test.go` to ensure correctness.
This subsystem primarily reads from SQLite schemas without modifying any core tables directly.
Goroutine synchronization guarantees that standard output remains ungarbled across threads.
Database locks encountered during execution indicate activity from concurrent agents.
We maintain a strict zero CGo dependency policy for maximal cross-platform compatibility.
Future iterations will refine the command-line flags to offer structured output caps.

These constraints limit memory explosion when querying tables that store gigabytes of embedded vectors.
Overall, the query tools function as diagnostic read-only interfaces rather than state mutators.
