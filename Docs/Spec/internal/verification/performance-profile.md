# Performance Profile: internal/verification

## LLM Latency Costs
Because LLMs evaluate the raw string generated by a prior LLM string, the `internal/verification` loop effectively doubles the base latency of every task attempt. If an issue triggers a retry (`VerifyWithRetry`), costs balloon multiplicatively.

## String Allocation Controls
To mitigate memory spikes due to potentially massive generation blocks accumulating in memory, the system uses rigid truncation logic internally:
- `truncateForVerification` clamps code chunks to `8000` characters to bound the input context provided to the LLM.
- `truncateContext` explicitly limits specialized knowledge extraction context limits at `2000` characters per query iteration. 

## Concurrency
The struct enforces thread safety locally via its `sync.RWMutex`, wrapping injection variables (`SetTaskExecutor` / `SetSessionContext`) guaranteeing safety under simultaneous multi-agent task spans.
